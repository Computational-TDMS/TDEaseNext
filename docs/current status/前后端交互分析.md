# 前后端交互逻辑分析

## 📋 当前架构分析

### 1. 工作流保存流程

**前端流程**：
```
用户修改工作流 → saveWorkflow() → createWorkflow API → 后端创建新工作流
```

**问题**：
- `saveWorkflow` 方法总是调用 `createWorkflow` API
- 即使工作流已存在（如 `wf_test_full`），也会创建新的工作流ID
- 导致测试工作流每次保存都产生新版本

**后端流程**：
```
POST /api/workflows → create_workflow() → 生成新ID → 保存到数据库
PUT /api/workflows/{id} → update_workflow() → 更新现有工作流
```

### 2. 工作流执行流程

**前端流程**：
```
executeWorkflow() → saveWorkflow() (可选) → executeCompiled() → POST /api/workflows/execute
```

**后端流程**：
```
POST /api/workflows/execute → 
  1. 从 metadata.id 获取 workflow_id
  2. 检查工作流是否存在，不存在则创建
  3. 创建新的 execution_id (正确)
  4. 执行工作流
```

**问题**：
- 执行时，如果工作流不存在，后端会创建新工作流（这是正确的）
- 但是，如果前端每次保存都创建新的工作流ID，那么测试工作流就会有很多个版本

### 3. 测试工作流问题

**当前行为**：
1. 前端加载 `wf_test_full`
2. 前端调用 `saveWorkflow()` → 创建新的工作流ID（错误！）
3. 执行时使用新的工作流ID

**期望行为**：
1. 前端加载 `wf_test_full`
2. 前端调用 `saveWorkflow()` → 检查工作流是否存在
   - 如果存在（如 `wf_test_full`），则更新
   - 如果不存在，则创建
3. 执行时使用同一个 `wf_test_full` ID
